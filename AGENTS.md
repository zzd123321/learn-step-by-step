# 学习仓库长期规范

本仓库用于按步骤学习全栈与 AI 应用开发。所有协作者和自动化助手都必须遵守以下规则。

## 内容语言

- 所有学习内容使用中文编写。
- 术语可以保留英文原文，但必须配套中文解释。

## 目录边界

- 每个学习方向独立维护，不跨目录修改其他方向文件。
- 修改某条学习路线时，只处理该路线目录内的课程、示例和进度文件。
- 根目录规范文件只在初始化、规则调整或跨路线协调时修改。
- 各线程不应同时写入同一个文件，避免覆盖彼此进度。

## 教学节奏

- 每次教学只推进一个可消化的知识点或小模块。
- 不追求一次覆盖大量概念，优先确保能理解、能运行、能复盘。

## 每节课内容要求

每节课不使用冗长固定模板，采用类似 Vue、TypeScript、Java 官方学习文档的写法：围绕一个知识点，把概念、示例和细节讲清楚。

每节课必须重点保证：

1. 知识点准确：定义、适用场景、关键规则和边界条件必须清楚；涉及语法、API、框架行为时，优先参考官方文档、语言规范或权威资料。
2. 代码详细解析：示例代码要尽量小而完整，并解释关键语句、执行流程、参数含义、常见误区和真实项目中的用法。

可按需要补充运行方式、验证结果、常见错误、练习或下一步建议；不要为了凑格式写空泛内容。

## 验证规则

- 不得伪造代码运行结果。
- 只有真实执行过的命令才能标记为“已验证”。
- 如果命令未执行、执行失败或受环境限制，必须如实说明。

## 安全与仓库卫生

- 不得提交密钥、Token、`.env`、个人配置、`node_modules`、`venv`、构建产物或数据库数据文件。
- 不得提交本地缓存、临时文件、IDE 私有配置或与学习内容无关的产物。

## Git 操作规则

- 提交前必须执行 `git diff --check`。
- 暂存时必须使用精确路径 `git add <path>`，不得使用 `git add .`。
- 提交完成后推送到当前分支对应远程。
- 若当前分支没有 upstream，则使用 `git push -u origin 当前分支`。
- 推送失败时，必须明确说明失败原因、已完成的本地提交以及下一步所需操作，不得假装推送成功。
- 不得执行 `git reset --hard`、`git clean`、`git push --force`、`git commit --amend`，除非用户明确要求且已确认风险。
